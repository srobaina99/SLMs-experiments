Implemented — phase2 kvl_beam

KVL-Scored Real Beam Search

Token-by-token beam search that ranks partial hypotheses by learner vocabulary difficulty (KVL/GLMM scores) instead of post-hoc A1-ratio reranking over independent samples. Used as a readability / vocabulary intervention in SLMs-experiments (primary binary outcome is an automated readability proxy, not a CEFR proficiency test).

Deprecated best-of-N vs live KVL beam

The project evaluates small language models on whether interventions produce simpler English for beginner learners. The old phase2 beam command used BeamSearchGenerator in src/slm_experiments/models/beam.py — but despite the name, it is not real beam search. It drew N independent full-length samples (best-of-N), then reranked finished answers by A1 vocabulary ratio. At temperature=0.0 that path is deprecated and hard-fails in the CLI.

The live KVL beam decoder (phase2 kvl_beam) performs genuine beam search: at each token step it expands the top-k logits for each active hypothesis, keeps the best W partial sequences ranked by a running mean GLMM score from KvlLookup, and only evaluates KVL when a content word completes at a word boundary. This mirrors the manual llama.cpp eval loop used by guided decoding in docs/guided-decoding.md, but optimizes for continuous difficulty scores rather than a binary A1 token filter. Stopping (intentional first-finish): returns the first finished non-empty candidate so answers can complete naturally — mean KVL does not reward EOS, so max-KVL-at-length would pad to max_new_tokens (see docs/kvl_beamsearch.md).

Best-of-N + post-hoc rerank

  • Generate N complete answers independently
  • Score finished text with A1-ratio or logprob
  • No interaction between samples during decoding
  • Void at temperature 0 — CLI hard-fails
Live — KVL beam

Token-by-token beam search

  • Maintain W partial hypotheses in parallel
  • Update KVL running mean at word boundaries
  • Prune to top-W by KVL score (+ logprob tiebreak)
  • Manual eval loop via llama.cpp (like guided decoding)

End-to-End Flow

From CLI invocation through evaluation. KVL metrics on the final answer reuse the existing compute_kvl_metrics path for reporting; the decoder uses the same lookup at decode time for pruning.

Beam State Data Structure

Each active hypothesis is a KvlBeamCandidate. Unlike the current BeamCandidate (token IDs + text only), KVL beam tracks partial-word state and incremental difficulty scores. 

@dataclass
class KvlBeamCandidate:
    token_ids: list[int]           # full sequence including prompt continuation
    text: str                      # decoded UTF-8 so far
    cumulative_logprob: float      # sum of log P(token | context)
    pending_word: str              # incomplete BPE word being assembled
    completed_content_words: list[str]  # lemmatized/cleaned words scored so far
    kvl_scores: list[float]         # GLMM score per completed content word
    finished: bool                 # hit stop sequence or max length

@dataclass
class KvlBeamState:
    beams: list[KvlBeamCandidate]  # current top-W hypotheses
    beam_width: int                # W — number kept after each step
    expand_top_k: int              # logits considered per beam per step
    step: int                      # decode step counter

After each expansion, all candidate children are scored, sorted by kvl_running_mean() (descending), then by cumulative_logprob, and truncated to beam_width. Finished beams remain in a separate pool and compete for the final selection.

Scoring Function

KVL beam ranks partial hypotheses by the same aggregate the evaluation pipeline reports: mean GLMM score over looked-up content words. Function words are excluded via the existing TextEvaluator.extract_content_words() logic.

kvl_running_mean = Σ scores(wi) / |{ wi : lookup(wi) ≠ None }|

rank(beam) = (kvl_running_mean, cumulative_logprob) # lexicographic, both descending

OOV handling options

Exclude (default)

OOV words do not enter the mean denominator — matches compute_kvl_metrics v1 behavior. Beams with more known easy words rank higher.

Penalty constant

Assign OOV words a fixed score (e.g. −3.0, below KVL_HARD_THRESHOLD = −1.0) so unknown words actively hurt the beam.

Coverage gate

Require lookup_coverage ≥ 0.5 before KVL ranking applies; otherwise fall back to logprob-only until enough words are scored.

Logprob tiebreaker

When two beams share the same running KVL mean (common early in decoding), break ties with cumulative_logprob so the model’s natural language prior still matters. Optional length normalization: divide logprob by len(token_ids) ** length_penalty to discourage overly long easy-word chains.

Known Limitations

📖

Lemma-only lookup

KvlLookup keys are lowercased surface lemmas (~6.8k words). Inflected forms like “friends” or “running” are OOV unless they appear in the lookup. v1 excludes OOV from the running mean, which can inflate scores for beams that avoid inflection.

🧩

Tokens ≠ words (BPE subwords)

Decoding is token-level; KVL is word-level. A content word may span multiple BPE tokens (“some” + “one”), and a single token may prefix multiple words (“run” in “run”, “running”, “runner”). A word_tracker module must buffer partial tokens and detect boundaries — same core problem as guided decoding’s A1TokenIndex.

⚡

Compute cost

Real beam search with width W and expand-top-k requires O(W × k) forward passes per token — far more expensive than one-shot generation or best-of-N. A W=4, k=10 decode of 100 tokens ≈ 4,000 eval steps vs. ~100 for greedy. Practical sweeps need smaller grids than the current beam width sweep.

🎯

Proxy vs. pass criteria

KVL GLMM scores measure estimated learner receptive knowledge, not whether output meets A1 readability thresholds (FK ≤ 5, Fog ≤ 6, Spache ≤ 4). Optimizing KVL during decode may not monotonically improve meets_a1_criteria — empirical comparison against best-of-N and guided decoding is required.

🔧

llama.cpp manual eval loop

Current wrappers use one-shot llm(prompt, max_tokens=…). KVL beam requires a new per-step eval API (as planned for ConstrainedDecoder). This is the highest-risk integration point and should be spiked on one GGUF model before a full sweep.

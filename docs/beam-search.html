How Beam Search Works

A visual guide to the decoding algorithm that balances quality and speed when language models generate text one token at a time.

The problem: choosing the next word

At each step, a language model outputs a probability for every token in its vocabulary. The challenge is deciding which sequence of tokens to produce — not just the single highest-probability token at each step.

Greedy

Pick the best token each step

Fast, but can miss globally better sequences. Like always taking the steepest downhill path and getting stuck in a local minimum.

Beam search

Keep top-k partial sequences

Maintains k promising hypotheses in parallel. Prunes the rest. A sweet spot between greedy and exhaustive search.

Exhaustive

Explore every path

Guarantees the best sequence, but is intractable — with 50,000 tokens and 20 steps, that's 50,00020 paths.

The idea in one picture

With beam width k = 2, only the 2 highest-scoring partial sequences survive at each depth. All others are pruned (grey).

How sequences are scored

Each partial sequence gets a cumulative log-probability — the sum of log-probabilities of every token chosen so far. Higher (less negative) is better.

score("The cat") = log P(The | prompt) + log P(cat | prompt, The)

Because probabilities multiply (and shrink quickly), we work in log-space to avoid underflow and to turn products into sums.

The algorithm

# beam_width = k, max_steps = number of tokens to generate def beam_search(prompt, model, k, max_steps): beam = [(prompt, 0.0)] # list of (sequence, cumulative_log_prob) for step in range(max_steps): candidates = [] for (seq, score) in beam: for token, log_prob in model.next_token_probs(seq): candidates.append((seq + token, score + log_prob)) # Sort by score descending, keep top k candidates.sort(key=lambda x: x[1], reverse=True) beam = candidates[:k] if all_finished(beam): break return beam[0] # highest-scoring complete sequence

Key takeaways

  • Beam width k controls the quality/speed tradeoff. Larger k explores more but costs k× more compute per step.
  • Pruning happens after every expansion — only the k best cumulative scores survive.
  • Not guaranteed optimal — a sequence pruned early can never come back, even if it would have been best overall.
  • Length bias — longer sequences accumulate more negative log-probs. Production systems often normalize by length (length penalty).
  • Used in machine translation, speech recognition, and any seq2seq task where you want better output than greedy decoding without exhaustive search.
